# Data Generation Strategy: Iterative Teacher Improvement

## Goal

Produce a high-quality labelled dataset for fine-tuning `llama3.2:3b` (the student),
by first iteratively improving `qwen2.5:32b` (the teacher) using frontier-model correction.

---

## Two-Phase Plan

```
Phase 1: Improve the teacher
  Iteratively fine-tune qwen2.5:32b using frontier-corrected labels
  → qwen2.5:32b v+n: a reliable, specialised data factory

Phase 2: Train the student
  Use qwen2.5:32b v+n to generate bulk training data
  → Fine-tune llama3.2:3b
```

The student is only trained once Phase 1 converges. All iterative cost is spent improving the teacher, not the student.

---

## Training Format and Expectation

|                   | qwen2.5:32b (teacher)                | llama3.2:3b (student)          |
|-------------------|--------------------------------------|--------------------------------|
| system message    | full template                        | one-liner                      |
| user message      | webtext                              | webtext                        |
| assistant message | perfect summary                      | perfect summary                |
| goal              | be able to create good training data | be able to create good summary |
| what it does      | learn the template and expectation   | learn the expectation          |
| specialised       | lock to one output format            | lock to one output format      |

---

## Phase 1: Teacher Improvement Loop

Each cycle adds 500 new webtexts. Labels from previous cycles are reused — only new rows are generated and corrected each cycle.

| Cycle | generation input         | Who generates labels | output                 | Who corrects   | output (to keep)     | input                  | model to fine tune | result (keep last) |
|-------|--------------------------|----------------------|------------------------|----------------|----------------------|------------------------|--------------------|--------------------|
| 1     | webtext 1 - 500 rows     | qwen2.5:32b          | v1 imperfect summaries | frontier model | v1 perfect summaries | 500 perfect summaries  | qwen2.5:32b        | qwen2.5:32b v+1    |
| 2     | webtext 501 - 1000 rows  | qwen2.5:32b v+1      | v2 imperfect summaries | frontier model | v2 perfect summaries | 1000 perfect summaries | qwen2.5:32b v+1    | qwen2.5:32b v+2    |
| 3     | webtext 1001 - 1500 rows | qwen2.5:32b v+2      | v3 imperfect summaries | frontier model | v3 perfect summaries | 1500 perfect summaries | qwen2.5:32b v+2    | qwen2.5:32b v+3    |
| ...   | ...                      | ...                  |                        |                |                      |                        |                    | ...                |
| n     | ...                      | ...                  | vn perfect summaries   |                |                      |                        |                    | qwen2.5:32b v+n    |

**Why reuse previous cycle labels rather than regenerating:**
Rows from earlier cycles were already corrected to "perfect" by the frontier model.
Regenerating them with a newer qwen version would cost frontier model tokens for no
quality gain — the labels are already at the ceiling the frontier model can provide.

**Why cumulative training (not per-cycle only):**
Fine-tuning only on the latest 500 rows causes catastrophic forgetting of earlier
examples. Training on all accumulated data each cycle is essential. QLoRA cost on
1000–3000 rows is small.

### Notes about Prompt Template for Teacher

- Teacher: Train with IS_MANUFACTURING, MANUFACTURING_PROCESSES, TECHNICAL_CAPABILITIES
- Student: If these are not needed, remove them from the training data generated by teacher

| Template                                    | Used for             |

| `prompt_template__02__plus_capabilities.md` | qwen2.5:32b training |

The capabilities template (`_02`) is a superset of `_01`. Training on `_02` means
the model also handles the `_01` field set, since all `_01` fields are present.

## Verification Procedure and Stopping Criteria

### Step 1: Establish a trusted head corrector (one-time)

```
qwen2.5:32b output
        │ (template, input, output) - agree? if not then correct
        ▼
  ┌─────────────┐     corrected result     ┌──────────────────────┐
  │  Kimi K2.5  │ ───────────────────────► │ Claude / Deepseek    │
  │  (corrects) │                          │ (verifies correction)│
  └─────────────┘                          └──────────────────────┘
                                                      | (qwen gives A, kimi corrects it to B) - agree?
                                                      +-- many yes -> Kimi trusted as head corrector
                                                      +-- many no  -> pick a different head corrector
```

Run this on a sample of records before committing to Kimi as the corrector for all cycles.
If multiple frontier models consistently agree with Kimi's corrections, Kimi is trusted.

### Step 2: Per-cycle correction and stopping signal

```
qwen2.5:32b output
        │ (template, input, output) - agree? if not then correct
        ▼
  ┌─────────────┐     corrected result     ┌──────────────────────┐
  │  Kimi K2.5  │ ───────────────────────► │ Claude / Deepseek    │
  │  (corrects) │                          │ (verifies correction)│
  └─────────────┘                          └──────────────────────┘
                                                      | (qwen gives A, kimi corrects it to B) - agree?
                                                      +-- yes -> keep training record
                                                      +-- no  -> option 1: important record -> further verification
                                                              -> option 2: says result wrong -> ignore record
                                                              -> option 3: says qwen is better -> ignore record
```

When Kimi consistently finds little or nothing to correct across a sample of outputs,
stop cycling. The teacher is ready to generate bulk training data for `llama3.2:3b`.

## Phase 2: Student Training Loop

Data generation for the student uses the same teacher correction pipeline (Step 2 above).
The label generator is now fixed at qwen2.5:32b v+n — it no longer changes between cycles.

| Cycle | input (created by teacher)     | model to fine-tune  | result          | stable? |
|-------|--------------------------------|---------------------|-----------------|---------|
| 1     | 500 perfect summaries          | llama3.2:3b         | llama3.2:3b v+1 | check   |
| 2     | 1000 perfect summaries         | llama3.2:3b v+1     | llama3.2:3b v+2 | check   |
| 3     | 1500 perfect summaries         | llama3.2:3b v+2     | llama3.2:3b v+3 | check   |
| ...   | (not enough perfect summaries) |                     | ...             | ...     |
| n     |                                | llama3.2:3b v+(n-1) | llama3.2:3b v+n | stop    |

Not enough perfect summaries:

- If we don't have enough perfect summaries generated and used for training teacher model
- Then use the teacher `Step 2: Per-cycle correction and stopping signal` procedure to create more data

**Stable**

- Means further training cycles no longer improve output quality.
- The growing dataset of verified perfect summaries is a valuable asset regardless of how many cycles are needed — it can be reused for future student models.
