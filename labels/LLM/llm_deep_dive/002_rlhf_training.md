# Reinforcement Leaning with Human Feedback

The Standard RLHF Process

## Phase 1: Supervised Fine-Tuning (SFT) (Optional but common)

Before even starting with RLHF, a base pre-trained Large Language Model (LLM) is often fine-tuned on a smaller, high-quality dataset of curated
prompt-and-response pairs. This adapts the model to the desired style and domain (e.g., instruction-following, summarization).

```
   [Pre-trained LLM]
          |
          v
[High-Quality Prompts & Responses]
          |
    (Fine-Tuning)
          |
          v
   [SFT/Policy Model v0]
```

## Phase 2: Training the Reward Model (RM)

The goal here is not to generate better text, but to create a model that can score a given response for a given prompt, mimicking human judgment.

1. Data Collection: For a single prompt, several responses are generated by the SFT model.
2. Human Ranking: A human labeler ranks these responses from best to worst.
3. Training Data Creation: The ranking is converted into a dataset of comparison pairs. For example, if the ranking is Response D > B > A > C,
   you create pairs like (Prompt, D, B), (Prompt, D, A), (Prompt, B, A), etc., where the first response is better than the second.
4. RM Training: The Reward Model (which is often a modified version of the SFT model) is trained on this comparison data. It learns to assign a
   higher score to the preferred response in each pair. Its output is a single scalar value (a score), not text.

```
                  (Prompt)
                     |
         [SFT/Policy Model v0]
                     |
      /      |       |      \
((Response A)) ((Response B)) ((Response C)) ((Response D))
                     |
            (Human Annotation)
                     |
          (Ranked: D > B > C > A)
                     |
 (Create Comparison Pairs: (D,B), (D,C), (D,A), (B,C), etc.)
                     |
                    ===> Train a new model <===
                     |
                  [Reward Model (RM)]
                     |
                  (Output: a scalar "goodness" score)
```

### Conceptual Data

| prompt | R_A | R_B | R_C | R_D | Rank          |
|--------|-----|-----|-----|-----|---------------|
| (text) | ... | ... | ... | ... | D > B > C > A |

or

| prompt | R_A | R_B | R_C | R_D | pair_1 | pair_2 |                             |
|--------|-----|-----|-----|-----|--------|--------|-----------------------------|
| (text) | ... | ... | ... | ... | (D,B)  | (D,C)  | ...  <-- One Complex Record | 

### Presented Data

| Prompt | Chosen Response | Rejected Response |                        |
|--------|-----------------|-------------------|------------------------|
| (text) | Full Response D | Full Response B   | <-- Training Example 1 |
| (text) | Full Response D | Full Response C   | <-- Training Example 2 |

Note: How it works: The training loop is incredibly simple. The data loader just reads one row, and that row contains everything needed for one training step

## Phase 3: Reinforcement Learning (RL) with PPO

This is the final and most complex phase. Here, the Reward Model is used as a "reward function" to improve the SFT model (now called the "Policy Model").

1. The Loop Begins: The Policy Model is given a new prompt from the dataset.
2. Generate: It generates a response.
3. Score: The Reward Model looks at the prompt and the generated response and assigns it a score (the "reward").
4. Update: A reinforcement learning algorithm (typically Proximal Policy Optimization, or PPO) uses this reward score to calculate a policy update.
   This update nudges the weights of the Policy Model in a direction that is more likely to produce high-reward responses in the future.
5. Constraint: A penalty (KL divergence) is applied to ensure the Policy Model's output doesn't stray too far from the original SFT model,
   preventing it from generating nonsensical text just to "hack" the Reward Model.

```
               +---------------------------------- [Reference Model (frozen copy of original policy)]
               |                                                   |
               | (calculates penalty for deviating too much)       |
(entry)        v                                                   |
[Prompt Dataset] ---------------------------------------------+    |
     |                                                        |    |
     v                                                        |    |
[Policy Model] --generates--> [Response]                      |    |
     |                           |                            |    |
     |                           v                            |    |
     |                   [Reward Model] --scores--> [Reward]  |    |
     |                           |                            |    |
     +----->----[PPO Algorithm]--+----------------------------+    |
                    │      ^                                       |
                    │      +----(gets final update signal)---------+
                    v
[Policy Model (updated)] --becomes the new policy for the next loop-->  

Note:
* Reference Model: This is a frozen copy of the Policy Model from before Phase 3 started. It doesn't get updated.
* KL-divergence penalty: If the new model starts to become too different from the original, it gets a penalty. This keeps the model "on-leash"
```

### Steps - Misconception

You may imagine that the Reinforcement Learning procedure is the same as Phase 2, and substitute the human step with a machine step like so:

1. Generate multiple responses (A, B, C, D).
2. RM ranks them. (D > B > C > A)
3. Create a `|prompt|chosen|rejected|` table from the ranking.
4. Fine-tune the main model on this new table.
5. Loop.

This sometimes is called "RL through AI Feedback" (RLAIF) - This is not a standard procedure

### Steps - Actual

1. Get a Prompt: A single prompt is sampled from the dataset.
2. Generate a Response: The Policy Model (the one we're trying to improve) generates a single response to the prompt.
3. Get a Reward: The Reward Model takes the (prompt, response) pair and outputs a single scalar score (e.g., 4.7). This score is the "reward." It
   doesn't rank multiple options; it just rates the one it sees.
4. Update the Policy: The RL algorithm (PPO) uses this single reward score to calculate an update for the Policy Model's weights. The update
   nudges the model to be slightly more likely to produce responses that would get a high score from the RM. This is not "fine-tuning" on a
   static dataset; it's a direct, gradient-based update from a live signal.
5. Loop: Repeat with a new prompt.
